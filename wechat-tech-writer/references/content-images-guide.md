# 内容配图生成指南

> **核心原则**：宁缺毋滥，质量优于数量。只在真正需要时才生成内容配图，不设硬性数量上限，但**每个 H2 章节最多 1 张**。
>
> **本指南只讲"判断要不要配 + 通用质量规范"**。具体的 Type/Style 选型、prompt 模板请看 `image-styles/` 目录的四个文件：
>
> - `image-styles/styles.md` —— 12 Type × 9 Style 风格库
> - `image-styles/auto-selection.md` —— 根据内容信号自动选 Type+Style
> - `image-styles/style-presets.md` —— 按文章类型预置组合
> - `image-styles/prompt-construction.md` —— 每种 Type 的专用 prompt 模板

---

## 一、判断是否需要内容配图

### 决策流程

```
阅读文章内容
  │
  ├─ 有明确数据对比？     → comparison（对比说明）
  ├─ 有复杂技术架构？     → architecture（架构图）
  ├─ 有抽象概念/原理？    → concept（概念讲解）
  ├─ 有教程/操作步骤？    → process    ⚠️ 不用 ComfyUI，改用 Mermaid 或跳过
  ├─ 有知识科普/课程感？  → whiteboard ⚠️ 不用 ComfyUI，改用 Mermaid 或跳过
  ├─ 有数据/统计/趋势？   → data-viz（数据可视化）
  ├─ 有知识框架/全景？    → mindmap（思维导图）
  ├─ 有发展历程/路线图？  → timeline（时间线）
  ├─ 有 Tips/要点列表？   → checklist（清单图）
  ├─ 有核心金句/观点？    → quote-card（金句卡片）
  ├─ 有典型使用场景？     → scene（场景还原）
  ├─ 有情感/氛围渲染？    → atmosphere（氛围烘托）
  └─ 其他情况             → 不生成
```

> **⚠️ 关于 process 和 whiteboard**：这两类用 ComfyUI 出图效果普遍很差（手写体/步骤编号容易糊、字会扭曲）。如果文章确实需要流程或板书，改用 Mermaid 等代码画图工具，或直接跳过不配。

### 应该生成的典型场景

| 场景信号 | 推荐 Type | 例子 |
|---------|-----------|------|
| 多产品 / 多模型的指标对比 | comparison / data-viz | "Claude vs GPT vs Gemini 响应速度对比" |
| 系统分层、模块关系、数据流 | architecture | "VSCode AI 架构三层结构" |
| 抽象概念需要视觉隐喻 | concept | "什么是 Agent：从聊天到执行" |
| 知识体系、主题全景 | mindmap | "AI 大模型生态全景" |
| 时间维度的事件序列 | timeline | "LLM 发展史关键节点" |
| 可成清单的要点 | checklist | "调 Prompt 的 7 条 Tips" |
| 文章核心金句 | quote-card | 截图转发用 |

### 不应该生成配图的情况

1. **装饰性图片** —— 通用科技插画、人物剪影、与正文无关的背景图
2. **重复封面** —— 与封面图主题一致、没有新增信息
3. **简单内容** —— 3-5 项的小列表、单一步骤、只有 1-2 个数据点，文字更清晰
4. **场景插画类陈词滥调** —— "一个人在电脑前工作"、"团队开会"、"未来城市"

## 二、配图数量与密度

**不设硬性上限**，按需生成。原则：

- **每个 H2 章节最多 1 张**
- 没有可视化价值的纯文字段落不配
- 不确定要不要配的，**不配**
- 经验密度参考（仅供心理预期，不强制）：
  - 2000 字文章：1-3 张
  - 3000 字文章：2-4 张
  - 新闻、纯观点、方法论：常常 0 张

**封面图**单独处理，强制 1 张，由发布器通过 `--cover` 参数上传，**不嵌入正文**。

## 三、选 Type × Style

具体规则见 `image-styles/auto-selection.md` 和 `image-styles/styles.md`，要点：

1. **Type**（画什么）：12 种，按上面决策流程从内容信号匹配
2. **Style**（什么调子）：9 种
   - 正式专业 → 扁平矢量 / 蓝图 / 商务
   - 轻松幽默 → 手绘 / 温暖
   - 犀利观点 → 海报 / 极简
3. **没有特别要求时**，每种 Type 用 `styles.md` 末尾的"首选风格"即可
4. **同一篇文章的多张配图，保持 Style 一致**，不要每张换风格

---

## 四、构建 Prompt

每种 Type 都有专用 prompt 模板，见 `image-styles/prompt-construction.md`。这里只讲跨 Type 的硬要求：

1. **图内文字按场景选语言**
   - 面向中文读者的公众号文章，优先用中文
   - 代码、术语、品牌名（Claude、GPT、API、RAG 等）保留原文
   - 不要为了"统一中文"把品牌名生硬翻译
2. **直接写要显示的文字，不要 placeholder**
   - ❌ `[产品名1]`、`[指标名]`
   - ✅ `Claude Sonnet 4`、`响应时间 (ms)`
3. **用文章里的真实数据**，不要让模型自己编数字
4. **一图一个焦点**，不要塞太多元素
5. **抽象优于具象**
   - "AI 赋能" → 光线连接感、节点网络 ✅
   - 画一个机器人坐在桌前 ❌

## 五、生成执行

### 首选：本地 ComfyUI

```bash
python3 scripts/comfyui_gen.py \
  --prompt "你构建的完整提示词" \
  --workflow templates/image_z_image.json \
  --width 1024 --height 768 \
  --negative "blurry, low quality, deformed, distorted text"
```

脚本把生成图片的绝对路径打到 stdout，取最后一行在文章中引用即可。

**尺寸约定**：

| 用途 | 比例 | 尺寸参数 |
|------|------|---------|
| 封面图 | 2.35:1 | `--width 1024 --height 432` |
| 内容配图（全部 Type） | 4:3 | `--width 1024 --height 768` |

### 备选：Hermes 内置生图

ComfyUI 未启动时**不要静默退回**。先告诉用户"本地 ComfyUI 未在线"，让用户选：

1. 手动启动 ComfyUI 后重试
2. 改用 `image_generate(prompt=..., aspect_ratio="landscape")`（无需自备 API key）

## 六、嵌入文章

### 位置原则

**核心要求**：图片要出现在**与其内容相关的位置**，让读者读到那里时图能起到解释/印证/补充的作用，而不是孤立存在。

**两种合理的摆放方式**：

1. **放在引出它的段落之后**（适用于：一段文字铺垫好背景/数据，紧接着用图呈现）
2. **放在段落之间**（适用于：H2 章节有多段，图片承接前面已展开的内容，并引出后面更深入的讨论）

只要图与"前面已经讲过的内容"有逻辑衔接、读者能看懂它在那里的原因，就是合理位置。

**不应该的位置**：

- ❌ 直接贴在 H2 标题之后（读者还没读到相关内容，看图一头雾水）
- ❌ 章节结尾、与下一节的过渡处（与上下文都脱节）
- ❌ 把图集中堆在文章末尾"图集"区
- ❌ 单独开"图片说明"小节

### 示例 A：段落之后

```markdown
## VSCode 与 Cursor 的性能对比

在响应速度方面，VSCode 启动 0.8 秒，Cursor 启动 1.2 秒，VSCode 快 33%。
代码补全延迟方面，VSCode 平均 150ms，Cursor 180ms。

![性能对比图](performance-comparison.png)

*图：VSCode vs Cursor 启动 / 补全耗时对比*

从数据看，VSCode 在性能优化上已经追上 Cursor……
```

### 示例 B：段落之间（推荐用于多段章节）

```markdown
## VSCode AI 架构的分层设计

VSCode 把 AI 能力分成三层：UI 层、Agent 层、模型层。
UI 层负责所有用户可见的交互入口，包括编辑器侧栏的 Chat 面板和命令面板。

Agent 层是这次架构升级的重点。它把对话、任务执行、上下文管理拆成独立模块，
让不同的功能可以共享同一套调度逻辑。

![VSCode AI 三层架构](vscode-ai-architecture.png)

*图：VSCode AI 三层架构与数据流*

模型层做了适配器抽象，可以同时接 Claude、GPT、Gemini，
切换模型不需要改 Agent 层逻辑。这也是它扩展性的来源……
```

后一种形式更常见 —— 图把上半段的"分层概念"具象化，下半段顺势继续讲细节。

### 图注规范

```markdown
![图片描述](image.png)

*图X：简短的图片说明*
```

要求：

- 简洁，**不超过 15 字**
- 不要重复图里已有的信息
- 数据图可附数据来源

```
✅ *图 1：主流 AI 模型响应速度对比（单位：ms）*
❌ *图 1：这张图展示了 Claude、GPT、Gemini 三个模型在响应速度方面的详细对比，可以看出 Claude 最快*
```

### 文件命名

```
✅ performance-comparison.png
✅ architecture-diagram.png
✅ workflow.png
✅ before-after-comparison.png

❌ image1.png
❌ 图片_final_v2.png
```

### 大小控制

| 类型 | 目标 |
|------|------|
| 封面图 | 500-800 KB |
| 内容配图 | 200-500 KB |
| 单张上限 | < 1 MB |
| 整篇文章总和 | < 2 MB |

需要压缩时：

```bash
convert input.png -quality 85 -resize 1920x1080 output.png
```

## 七、检查清单

### 生成前

- [ ] 这张图是不是真有视觉价值（不是装饰）
- [ ] 选的 Type 跟内容信号对得上
- [ ] 准备好了准确的数据 / 名称 / 文字
- [ ] Style 与其他配图、封面图协调

### 生成后

- [ ] 数据准确，数值与文章一致
- [ ] 文字清晰（中文不糊、品牌名拼写正确）
- [ ] 元素不重叠，对比度足够
- [ ] 文件大小合理

### 嵌入后

- [ ] 位置合理：图出现在与其内容相关的段落附近（段落之后或段落之间均可）
- [ ] 不是贴在 H2 标题正下方，也不是孤立在章节结尾
- [ ] 有恰当图注
- [ ] 移动端可读(文字够大、对比度够)
- [ ] 同一篇文章风格统一

## 八、最佳实践

### ✅ 应该做的

1. **按需配图** —— 不为配图而配图
2. **每 H2 ≤ 1 张** —— 节制
3. **数据准确** —— 图里的数值必须跟正文一致
4. **风格统一** —— 同文风格不要换来换去
5. **位置合理** —— 图与"前面已讲过的内容"有逻辑衔接，可在段落之后或段落之间，不要孤立或贴在章节起止处

### ❌ 不应该做的

1. 装饰性插图填空
2. 把简单列表硬画成图
3. 一篇文章多张图各换一种 Style
4. 让模型自由发挥数字
5. 图片文字强制要求"全部中文"

### 自测三问

1. 遮住图片，只读文字 —— 能读懂吗？
   - 能 → 图可能不必要
   - 难懂 → 图有价值
2. 这张图和封面图的信息是否重复？
3. 这张图是不是只是为了让排版"看起来不那么单调"？

---

**记住**：内容配图不是必需的，但生成的每一张都必须有价值、准确、清晰，且与文章其他配图风格一致。
