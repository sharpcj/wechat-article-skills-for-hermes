# 内容配图指南（产品经理视角）

> **核心原则**：宁缺毋滥。大多数产品经理文章**不需要**内容配图，封面图 + 内容结构图（强制）已经够用；只在真正能放大表达力时再加。**不设硬上限，但每个 H2 章节最多 1 张。**
>
> **本指南只讲"判断要不要配 + 通用规范"**。Type / Style 选型与 prompt 模板见 `image-styles/`：
>
> - `image-styles/styles.md` —— 12 Type × 9 Style 风格库
> - `image-styles/auto-selection.md` —— 根据内容信号自动选 Type+Style
> - `image-styles/style-presets.md` —— 按产品经理五类内容方向预置
> - `image-styles/prompt-construction.md` —— 每种 Type 的 prompt 模板
>
> **封面图**：见 `cover-image-guide.md`，发布时通过 `--cover` 单独上传，**不嵌入正文**。
> **内容结构图**：见 `structure-image-guide.md`，产品经理视角文章**强制 1 张**，与本指南讨论的"内容配图"是两回事。

---

## 一、判断是否需要内容配图

### 决策流程

```
阅读文章内容
  │
  ├─ 有明确数据 / 指标对比？        → comparison / data-viz
  ├─ 有产品架构 / 技术方案？        → architecture
  ├─ 有抽象概念需要解释？           → concept
  ├─ 有典型使用场景需要代入？       → scene
  ├─ 有知识框架 / 主题全景？        → mindmap
  ├─ 有产品演进路线 / 历程？        → timeline
  ├─ 有 Tips / 要点 / 注意事项？    → checklist
  ├─ 有核心金句 / 观点卡？          → quote-card
  ├─ 有教程 / 操作步骤？            → process    ⚠️ 不用 ComfyUI，改用 Mermaid 或跳过
  ├─ 有知识讲解 / 课程感？          → whiteboard ⚠️ 不用 ComfyUI，改用 Mermaid 或跳过
  └─ 其他情况                       → 不生成
```

> **⚠️ process 和 whiteboard**：ComfyUI 出图效果普遍较差（步骤编号、手写体容易糊），改用 Mermaid 等代码画图，或者跳过。

### 产品经理文章常见的"不必配图"情况

- **产品方法论**：观点为主，文字本身就是核心，硬配图反而稀释
- **行业观察**：分析为主，没明确数据时配图就是凑数
- **个人随笔型产品拆解**：第一人称叙述 + 真实场景描述，往往一张封面 + 一张内容结构图就够
- **凡是"我觉得加张图好像气氛比较好"** —— 不加

### 值得配图的产品经理文章场景

| 场景 | Type | 例子 |
|------|------|------|
| 多个 AI 工具的真实对比 | comparison | Dify vs Coze vs n8n 适用场景对比 |
| 我搭的方案 / 工作流的结构 | architecture | 我的 RAG 知识库架构 |
| 抽象产品概念解释 | concept | "Agent 比 Chat 多了什么" |
| 复盘里的演进历程 | timeline | 这款产品三个月的迭代轨迹 |
| 实操踩过的坑清单 | checklist | 调 Prompt 的 7 个常见坑 |
| 文章核心金句截图转发 | quote-card | 关键观点单图 |

## 二、配图数量与密度

**不设硬性上限**，按需生成。原则：

- **每个 H2 章节最多 1 张**
- 封面图 + 内容结构图（强制）已经是产品经理文的视觉骨架，**内容配图是锦上添花**
- 没有可视化价值的段落不配
- 不确定要不要的，**不配**

经验密度（不强制）：

| 文章类型 | 内容配图 |
|---------|---------|
| 产品方法论 | 0 张居多 |
| 行业观察 | 0-1 张 |
| 场景解决方案 | 1-2 张（架构 / 场景） |
| 效率提升实战 | 1-2 张（对比 / 流程结果） |
| 产品拆解 | 1-2 张（数据 / 架构） |

## 三、选 Type × Style

具体规则见 `image-styles/auto-selection.md` 和 `style-presets.md`。要点：

- **Type**（画什么）：12 种，按上面决策流程匹配
- **Style**（什么调子）：9 种
  - 产品经理文章常用：**温暖 / 扁平矢量 / 简约线条 / 商务**
  - 偏深度 / 严肃 → 商务 / 极简
  - 偏轻松 / 个人分享 → 温暖 / 手绘
  - 偏方法论 / 框架感 → 扁平矢量 / 简约线条
- **没有特别要求时**用 `styles.md` 末尾的 Type 默认首选
- **同一篇文章风格保持一致**，封面 + 内容结构图 + 内容配图三者调子要协调

---

## 四、构建 Prompt

模板见 `image-styles/prompt-construction.md`。跨 Type 硬要求：

1. **图内文字按场景选语言**
   - 面向中文读者的公众号文章，优先用中文
   - 品牌名（Dify、Claude Code、Coze、n8n、RAG、Agent 等）保留原文
   - 不要为了"全部中文"翻译品牌名
2. **直接写要显示的文字，不要 placeholder**
3. **用文章里的真实数据 / 真实产品名**，不要让模型自由发挥
4. **一图一个焦点**
5. **抽象优于具象** —— 不要动不动画机器人 / 西装人士

## 五、生成执行

### 首选：本地 ComfyUI

```bash
python3 scripts/comfyui_gen.py \
  --prompt "你构建的完整提示词" \
  --width 1024 --height 768 \
  --negative "blurry, low quality, deformed, distorted text"
```

**尺寸约定**：

| 用途 | 比例 | 尺寸参数 |
|------|------|---------|
| 封面图 | 2.35:1 | `--width 1024 --height 432` |
| 内容结构图 | 4:3 | `--width 1024 --height 768` |
| 内容配图（全部 Type） | 4:3 | `--width 1024 --height 768` |

### 备选：Hermes 内置生图

ComfyUI 未启动时**不要静默退回**。先告诉用户"本地 ComfyUI 未在线"，让用户选：

1. 手动启动 ComfyUI 后重试
2. 改用 `image_generate(prompt=..., aspect_ratio="landscape")`（无需自备 API key）

## 六、嵌入文章

### 位置原则

**核心要求**：图片要出现在**与其内容相关的位置**，让读者读到那里时图能起到解释 / 印证 / 补充的作用，而不是孤立存在。

**两种合理的摆放方式**：

1. **放在引出它的段落之后**（一段文字铺垫好背景或数据，紧接着用图呈现）
2. **放在段落之间**（H2 章节有多段时，图片承接前面已展开的内容，并引出后面更深入的讨论）

只要图与"前面已经讲过的内容"有逻辑衔接，读者能看懂它在那里的原因，就是合理位置。

**不应该的位置**：

- ❌ 直接贴在 H2 标题之后（读者还没读到相关内容）
- ❌ 章节结尾或与下一节的过渡处
- ❌ 集中堆在文章末尾"图集"区
- ❌ 单独开"图片说明"小节

### 示例 A：段落之后

```markdown
## 我是怎么对比这三款工具的

我在做客服知识库时，分别试了 Dify、Coze 和 n8n，从搭建速度、
可维护性、扩展性三个维度打分……

![三款工具对比](tools-comparison.png)

*图：Dify / Coze / n8n 三款工具实测打分*

可以看到，Dify 在……
```

### 示例 B：段落之间（推荐用于多段章节）

```markdown
## 我的 RAG 知识库是怎么搭起来的

最早的版本就是一个简单的"文档切片 + 向量检索 + LLM 回答"链路，
跑通是跑通了，但答错率很高，尤其在长文档里跨段落的问题上几乎不可用。

后来我把检索层重构了一遍，加了一层"问题改写 + 多路召回"，
召回阶段同时拿向量结果和关键词结果，再用模型重排序。

![我的 RAG 架构](rag-architecture.png)

*图：当前版本的 RAG 知识库架构*

这套架构跑下来，答错率从 30% 多降到了 8% 以下。下面我具体讲讲
每一层是怎么做的，以及踩过的几个坑……
```

后一种更常见 —— 图把上半段铺垫好的"我做了什么"具象化，下半段顺势继续讲细节。

### 图注

```markdown
*图X：简短说明（不超过 15 字）*
```

- 不重复图里已有信息
- 数据类图可附数据来源

### 文件命名

```
✅ tools-comparison.png
✅ rag-architecture.png
✅ workflow-result.png

❌ image1.png
❌ 截图_final_v2.png
```

### 大小

| 类型 | 目标 |
|------|------|
| 封面图 | 500-800 KB |
| 内容结构图 / 内容配图 | 200-500 KB |
| 单张上限 | < 1 MB |
| 整篇总和 | < 2 MB |

## 七、检查清单

### 生成前

- [ ] 真有视觉价值（不是凑气氛）
- [ ] 选的 Type 跟内容信号对得上
- [ ] 数据 / 产品名 / 文字都准备好了
- [ ] 跟封面 + 内容结构图的 Style 协调

### 生成后

- [ ] 数据准确，跟正文一致
- [ ] 文字清晰，品牌名拼写正确
- [ ] 元素不重叠，对比度够
- [ ] 文件大小合理

### 嵌入后

- [ ] 位置合理：图出现在与其内容相关的段落附近（段落之后或段落之间均可）
- [ ] 不是贴在 H2 标题正下方，也不是孤立在章节结尾
- [ ] 有恰当图注
- [ ] 移动端可读
- [ ] 整篇风格统一

## 八、最佳实践

### ✅ 应该做的

1. **节制** —— 产品经理文不靠图说话，靠观点和场景
2. **真实** —— 配图反映真实使用 / 真实数据
3. **统一** —— 封面、结构图、内容配图调子一致
4. **位置合理** —— 图与"前面已讲过的内容"有逻辑衔接，可在段落之后或段落之间，不要孤立或贴在章节起止处
5. **品牌名保留原文** —— 别翻译 Dify、Claude、Agent

### ❌ 不应该做的

1. 为方法论 / 行业观察文硬配图
2. 让模型编数字
3. 一篇文章多张图各换一种 Style
4. 图里堆机器人 / 西装人士做装饰
5. 强制要求"图里全部中文"

### 自测

- 遮住图片，只读文字，能读懂吗？能 → 图可能不必要
- 这张图和封面 / 内容结构图重复了吗？
- 这张图是不是只为"看起来不那么干"？

---

**记住**：产品经理文章的视觉骨架是封面 + 内容结构图（强制 1 张）。其他内容配图按需，每张都必须有价值。
